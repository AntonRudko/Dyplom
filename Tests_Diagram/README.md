# Tests_Diagram — бенчмарки продуктивності алгоритмів детермінізації

Папка містить 8 тестових сценаріїв, кожен з яких вимірює продуктивність алгоритмів детермінізації під конкретним кутом і генерує набір графіків у PNG.

---

## Запуск

```bash
cd Dyplom_Top

# Усі 8 базових тестів
python -m Tests_Diagram.run_all

# Усі epsilon-тести
python -m Tests_Diagram.run_all_epsilon

# Тільки конкретні тести (наприклад, 1 і 3)
python -m Tests_Diagram.run_all 1 3

# Окремий тест
python -m Tests_Diagram.test1_exponential_blowup

# Heatmap-порівняння
python -m Tests_Diagram.heatmap_comparison
python -m Tests_Diagram.heatmap_algorithmic
```

## Що вимірюється

Кожен тест (окрім test8) вимірює 4 метрики для кожного алгоритму:

| Метрика | Що показує |
|---------|------------|
| **Time** | Середній час детермінізації (секунди) |
| **Memory** | Пікове споживання RAM (КБ, через `tracemalloc`) |
| **DFA Size** | Кількість станів результуючого ДКА |
| **Operations** | Внутрішня метрика алгоритму (оброблені підмножини / ітерації / злиття / обчислені переходи) |

## nfa_generators.py — генератори тестових НКА

| Генератор | Опис | Параметри |
|-----------|------|-----------|
| `gen_nth_from_last(n)` | Класичний worst-case: мова «n-й символ з кінця = 1». НКА: n+1 стан → ДКА: рівно 2^n станів | `n` — позиція з кінця |
| `gen_dense_random(n, Σ, p)` | Густий випадковий НКА з гарантією зв'язності та недетермінізму | `n` — станів, `Σ` — алфавіт, `p` — ймовірність ребра |
| `gen_sparse_nfa(n, Σ, nd)` | Переважно детермінований НКА, лише `nd` частка переходів недетермінована | `nd` — частка недетермінізму |
| `gen_variable_alphabet(n, |Σ|, p)` | НКА зі змінним розміром алфавіту | `|Σ|` — кількість символів |
| `gen_epsilon_chain(n, ε)` | НКА з ε-переходами: базовий ланцюг + `ε` випадкових ε-зв'язків | `ε` — кількість ε-переходів |
| `gen_variable_nondet(n, Σ, nd)` | НКА зі змінною часткою недетермінованих переходів | `nd` — від 0.0 до 1.0 |
| `gen_multi_branch(n, k)` | НКА з k паралельними гілками довжини n, що зливаються в один приймаючий стан | `n` — довжина, `k` — гілок |

## Тест 1: Exponential Blowup (test1_exponential_blowup.py)

### Мета

Продемонструвати **найгірший випадок** детермінізації — коли НКА з n+1 станом неминуче перетворюється на ДКА з 2^n станів. Це теоретична нижня межа, яку неможливо обійти жодним алгоритмом.

### Параметри

- НКА сімейства «nth-from-last»: n = 4, 5, 6, 7, 8, 9, 10
- 5 повторів для усереднення часу

### Графіки (test1_exponential_blowup.png)

**4 субграфіки, вісь Y — логарифмічна шкала:**

1. **Time** — час детермінізації від n. Показує, що час зростає експоненційно для всіх алгоритмів. Навіщо: підтверджує теоретичну складність O(2^n) на практиці і дозволяє побачити константний множник кожного алгоритму.

2. **Memory** — піковий обсяг RAM від n. Показує, що пам'ять також зростає експоненційно (потрібно зберігати 2^n станів). Навіщо: демонструє практичну обмеженість — при n > 20 алгоритми вичерпають пам'ять.

3. **DFA Size** — кількість станів ДКА від n. Пунктирна лінія 2^n — теоретична межа. Показує, що всі алгоритми дають рівно 2^n станів. Навіщо: підтверджує, що жоден алгоритм не може зменшити ДКА для цього сімейства — навіть Brzozowski (який зазвичай дає мінімальний ДКА) дає 2^n, бо це вже мінімальний ДКА.

4. **Operations** — внутрішні операції від n. Показує різницю в кількості роботи: Brzozowski робить більше ітерацій (подвійна детермінізація + реверс), Subset та Lazy — однакову кількість підмножин. Навіщо: пояснює, чому один алгоритм повільніший за інший навіть при однаковому розмірі результату.

## Тест 2: Density Impact (test2_density_impact.py)

### Мета

Дослідити, як **щільність переходів** (ймовірність наявності ребра між двома станами) впливає на продуктивність. Щільніший НКА = більше недетермінізму = потенційно більший ДКА.

### Параметри

- 30 станів, алфавіт {0, 1, 2}
- Щільність: 0.02, 0.04, 0.06, 0.08, 0.10, 0.12, 0.15
- 3 випадкових НКА × 5 повторів на точку

### Графіки (test2_density_impact.png)

1. **Time** — час від щільності. Показує, як швидко зростає час кожного алгоритму при збільшенні кількості переходів. Навіщо: дозволяє визначити, який алгоритм краще справляється з густими НКА (багато недетермінізму), а який — деградує.

2. **Memory** — RAM від щільності. Показує, що при високій щільності пам'ять зростає (більше досяжних підмножин). Навіщо: важливо для практичного застосування — при яких щільностях алгоритм ще працює в розумних межах.

3. **DFA Size** — розмір ДКА від щільності. Показує, чи дає Brzozowski менший ДКА при високій щільності. Навіщо: обґрунтування вибору алгоритму — якщо потрібен мінімальний ДКА, Brzozowski може бути кращим вибором навіть якщо повільніший.

4. **Operations** — операції від щільності. Показує внутрішню вартість: Transset може робити менше операцій за рахунок злиття станів з однаковою поведінкою. Навіщо: пояснює чому алгоритм швидший/повільніший — це кількість роботи, а не лише час.

## Тест 3: Sparse Nondeterminism (test3_sparse_nondeterminism.py)

### Мета

Дослідити поведінку на **переважно детермінованих** НКА (лише 5% переходів недетерміновані). Це типовий випадок для НКА, побудованих з регулярних виразів — більшість переходів детерміновані.

### Параметри

- Розміри: 5, 8, 11, 14, 17, 20 станів
- 5% недетермінованих переходів
- Алфавіт {0, 1, 2}
- 3 НКА × 5 повторів

### Графіки (test3_sparse_nondeterminism.png)

1. **Time** — час від розміру НКА. Показує масштабування при мінімальному недетермінізмі. Навіщо: для майже-детермінованих НКА Subset перебудовує весь автомат з нуля, тоді як Lazy обчислює тільки потрібні переходи — тут видно цю різницю.

2. **Memory** — RAM від розміру. Показує що при малому недетермінізмі пам'ять зростає помірно. Навіщо: підтверджує, що для sparse НКА всі алгоритми працюють ефективно по пам'яті.

3. **DFA Size** — розмір ДКА від розміру НКА. Пунктир — розмір НКА (reference). Показує, що ДКА лише трохи більший за НКА (мало вибуху). Навіщо: демонструє, що при малому недетермінізмі детермінізація «майже безкоштовна» по кількості станів.

4. **Operations** — операції від розміру. Навіщо: показує внутрішню ефективність — Brzozowski робить зайву роботу (подвійна детермінізація) на НКА, що й так майже детерміновані.

## Тест 4: Branch Structure (test4_branch_structure.py)

### Мета

Дослідити поведінку на **структурованих** НКА з паралельними гілками. Стартовий стан недетерміновано обирає одну з k гілок. Subset Construction створює декартовий добуток позицій у гілках — O(n^k) станів.

### Параметри

- k = 3 гілки
- Довжина гілки: 2, 3, 4, 5, 6, 7
- 5 повторів

### Графіки (test4_branch_structure.png)

1. **Time** — час від довжини гілки (логарифмічна шкала). Показує поліноміальне зростання часу O(n^k). Навіщо: демонструє, що навіть без експоненційного вибуху детермінізація може бути дорогою — зростання n^3 при k=3 гілках.

2. **Memory** — RAM від довжини (логарифмічна шкала). Навіщо: показує кореляцію між кількістю станів ДКА та споживанням пам'яті.

3. **DFA Size** — розмір ДКА від довжини гілки. Пунктир — розмір НКА. Показує різницю між алгоритмами: Brzozowski може дати менший ДКА (мінімізація зливає симетричні гілки), Transset також зливає стани з однаковою поведінкою. Навіщо: це тест, де різниця у розмірі ДКА між алгоритмами **найбільш помітна** — ключове порівняння якості результату.

4. **Operations** — операції від довжини (логарифмічна шкала). Навіщо: показує, що Transset робить менше операцій ніж Subset за рахунок злиття еквівалентних станів, а Brzozowski — більше через подвійний прохід.

## Тест 5: Alphabet Size Impact (test5_alphabet_size.py)

### Мета

Дослідити вплив **розміру алфавіту** на продуктивність. Більший алфавіт = більше переходів на стан = більша вартість кожного кроку BFS.

### Параметри

- 15 станів, ймовірність ребра 0.04
- Алфавіт: |Σ| = 2, 3, 4, 6, 8 символів
- 2 НКА × 3 повтори

### Графіки (test5_alphabet_size.png)

1. **Time** — час від |Σ|. Показує зростання часу зі збільшенням алфавіту. Навіщо: дозволяє оцінити чутливість кожного алгоритму до розміру алфавіту. Brzozowski зростає найшвидше (|Σ| множить вартість кожного з 4 проходів), решта — лінійно.

2. **Memory** — RAM від |Σ|. Навіщо: більший алфавіт = більше переходів у ДКА = більше пам'яті. Показує, як ефективно кожен алгоритм використовує пам'ять.

3. **DFA Size** — розмір ДКА від |Σ|. Показує, що більший алфавіт збільшує кількість досяжних підмножин. Навіщо: пояснює, чому час зростає — не тільки через дорожчий BFS-крок, а й через більшу кількість станів.

4. **Operations** — операції від |Σ|. Навіщо: показує, що внутрішня вартість зростає пропорційно |Σ| — кожен стан потребує обробки по кожному символу.

## Тест 6: Epsilon Overhead (test6_epsilon_overhead.py)

### Мета

Виміряти **додаткові витрати** від обробки ε-переходів. Порівнює ε-варіанти алгоритмів при різній кількості ε-переходів у НКА.

### Параметри

- 15 станів, алфавіт {0, 1}
- Кількість ε-переходів: 0, 2, 5, 8, 12, 18
- 2 НКА × 3 повтори
- Алгоритми: Subset+ε, Brzozowski+ε, Transset+ε, Lazy+ε

### Графіки (test6_epsilon_overhead.png)

1. **Time** — час від кількості ε-переходів. Показує, наскільки дорожчим стає обчислення ε-замикання при збільшенні ε-зв'язків. Навіщо: дозволяє оцінити overhead ε-обробки для кожного алгоритму. При 0 ε — це чистий overhead самого механізму ε-closure (має бути мінімальний).

2. **Memory** — RAM від кількості ε. Навіщо: ε-переходи зливають стани і можуть як зменшувати (менше досяжних підмножин), так і збільшувати (ширше ε-замикання) споживання пам'яті.

3. **DFA Size** — розмір ДКА від кількості ε. Показує цікавий ефект: ε-переходи можуть зменшувати ДКА (зливають стани) або збільшувати (нові досяжні комбінації). Навіщо: демонструє, що ε-переходи — не просто overhead, а структурна зміна автомата.

4. **Operations** — операції від кількості ε. Навіщо: показує внутрішню вартість ε-closure — при великій кількості ε кожен крок стає дорожчим через BFS по ε-переходах.

## Тест 7: Nondeterminism Degree (test7_nondet_degree.py)

### Мета

Дослідити вплив **ступеня недетермінізму** на продуктивність. Базовий детермінований автомат поступово «засмічується» недетермінованими переходами.

### Параметри

- 18 станів, алфавіт {0, 1}
- Частка недетермінізму: 0%, 5%, 10%, 15%, 20%, 30%, 50%
- 2 НКА × 3 повтори

### Графіки (test7_nondet_degree.png)

1. **Time** — час від частки недетермінізму. Показує, як різко (або поступово) зростає час кожного алгоритму при збільшенні недетермінізму. Навіщо: ключовий графік для вибору алгоритму — якщо НКА переважно детермінований, Lazy може бути ефективнішим; якщо сильно недетермінований — Transset або Brzozowski можуть виграти.

2. **Memory** — RAM від частки nd. Навіщо: показує кореляцію між недетермінізмом і споживанням пам'яті.

3. **DFA Size** — розмір ДКА від частки nd. Пунктир — розмір НКА (reference). Показує, наскільки зростає ДКА при збільшенні недетермінізму. Навіщо: кількісно оцінює «ціну недетермінізму» в станах ДКА. При 0% ДКА ≈ НКА; при 50% ДКА може бути в рази більшим.

4. **Operations** — операції від частки nd. Навіщо: дозволяє побачити, що Subset завжди обробляє стільки ж підмножин (незалежно від nd), а Transset може зливати більше станів при низькому nd.

## Тест 8: Correctness Verification (test8_correctness.py)

### Мета

Верифікувати **коректність** усіх алгоритмів (4 базових + 4 ε-варіанти) на різноманітних НКА. Не вимірює продуктивність, а перевіряє, що всі алгоритми дають еквівалентні ДКА.

### Параметри

**Без ε (11 тестових НКА):**
- nth_from_last(5), nth_from_last(7)
- dense(20, p=0.1), dense(30, p=0.05)
- sparse(30, 5%), sparse(50, 10%)
- branch(4, k=3), branch(5, k=2)
- nondet(25, 0%), nondet(25, 20%), nondet(25, 50%)

**З ε (4 тестових НКА):**
- eps_chain(15, ε=3), eps_chain(15, ε=8)
- eps_chain(25, ε=5), eps_chain(25, ε=15)

Для кожного НКА генерується 100 випадкових слів і перевіряється, що:
1. Кожен ДКА приймає/відхиляє слово так само, як НКА
2. Усі ДКА між собою згодні

### Графіки (test8_correctness.png)

**2 субграфіки (стовпчикові діаграми):**

1. **State Blowup Ratio** — коефіцієнт вибуху (DFA Subset / NFA) для кожного тестового НКА. Показує, у скільки разів ДКА більший за НКА. Навіщо: візуалізує реальну «ціну» детермінізації для різних типів НКА. Найбільший blowup — на nth-from-last (теоретичний worst-case), найменший — на sparse НКА.

2. **Compression Ratios vs Subset** — коефіцієнт стиснення кожного алгоритму відносно Subset Construction (D_alg / D_subset). Значення < 1.0 означає менший ДКА. Навіщо: показує, наскільки ефективніше Brzozowski та Transset використовують стани порівняно з Subset. Для branch-структур різниця найбільш помітна — Brzozowski знаходить мінімальний ДКА, а Transset зливає симетричні гілки.
